## Systemic patterns (cross-cutting themes)

1. **Per-view ad-hoc timestamp parsing.** At least 6 confirmed findings reduce
   to the same root cause: each view/page constructs its own ISO8601 parser
   (fractional-only or whole-second-only) or buckets by UTC while neighbors use
   local time. One shared tolerant date helper would fix Popover source health,
   relative-time labels, AgentsView KPI, timeline grouping, and daily activity.
2. **Long work inside a 30s framed request.** The transport's hard 30s frame
   deadline silently overrides longer caller timeouts (600s migrations, 45s LLM
   calls, multi-minute regenerate-all), so the service keeps working while the
   client reports failure. Needs async command + event-channel progress.
3. **`sessions_fts.session_id` is UNINDEXED.** Every per-session FTS lookup is
   a full corpus scan (832 MB live DB): service keyword search (measured 27s),
   AI context reads, title-context batches.
4. **The AI generation pipeline is largely inert by default.** The Settings UI
   advertises Ollama/Anthropic/Gemini, auto-summary, auto-titles, sampling
   knobs; the service implements only OpenAI-protocol with an API key, reads
   none of the auto/sampling keys, regenerate-all matches zero sessions, and
   re-indexing clobbers AI summaries.
5. **The resume/launch surface (newest feature) fails end-to-end in the
   shipped configuration**: wrong codex invocation, missing Apple Events
   entitlement, Ghostty arg semantics, launchd-minimal PATH discovery, empty-cwd
   `open ''`, and all launcher failures swallowed.
6. **Ingestion durability gaps**: 2-day rescan window permanently skips older
   modified sessions; one bad line/oversized file drops a whole session;
   deterministic parse failures retry forever; FTS rebuild can never finalize;
   poison jobs starve the queue head.
7. **Docs promise what the product doesn't do** (index-time redaction, Windsurf
   full support, read-only source paths, FTS_VERSION) — high-trust claims that
   would mislead users and engineers.
8. **Tests that can't fail**: source-string-matching invariant tests,
   tautological smoke asserts, fixture DB generated by the deprecated TS schema,
   hand-rolled schema replicas.

## Confirmed findings — HIGH (26)

#### H01. Fixed 30s whole-frame deadline silently overrides the 10-minute migration command timeouts

- **Location**: `macos/Shared/Service/UnixSocketEngramServiceTransport.swift:252`
- **Found by**: service-runtime (round 1) — confidence high
- **Severity**: high
- **Verification**: confirmed by 2 adversarial lenses
- **Merged duplicates**: mcp-surface#0

readFrame() arms a hard-coded 30s wall-clock deadline (maximumFrameDurationSeconds) for the whole response frame, checked before each blocking syscall. The per-request timeout passed to send() only sets SO_RCVTIMEO/SO_SNDTIMEO. EngramServiceClient sends projectMove/projectArchive/projectUndo/projectMoveBatch with migrationCommandTimeout = 600s (EngramServiceClient.swift:8,140-153), but the client blocks in readFrame while the server runs the handler; if the response arrives later than ~30s after the request, the first readExact(count: length) iteration (or the second prefix iteration) hits the stale deadline and throws 'exceeded the per-frame deadline'. The service meanwhile keeps running the migration and commits it — the app/MCP reports failure for an operation that actually mutated session paths on disk and in the DB, inviting a user retry of an already-applied move. EngramMCP uses the same client, so MCP project_move has the same cap. The only test for this (EngramServiceIPCTests.swift:584 testTransportTracksWholeFrameWallClockDeadline) asserts the SOURCE CODE contains the identifier strings, not the behavior, so the timeout-cap regression is invisible to the suite.

> Evidence: `static let maximumFrameDurationSeconds: TimeInterval = 30  (line 13)
let deadline = Date().addingTimeInterval(maximumFrameDurationSeconds)  (line 252, readFrame)
--- EngramServiceClient.swift:8,140 ---
private static let migrationCommandTimeout: TimeInterval = 10 * 60
try await command("projectMove", payload: request, timeout: Self.migrationCommandTimeout)`

**Recommendation**: Thread the caller's request timeout into readFrame as the frame deadline (e.g. deadline = max(timeout, 30s)), keeping the 30s budget only as an inter-byte/trickle bound, and add a behavioral test that a response delayed >30s with a longer requested timeout still succeeds.

#### H03. Observability panels can never see EngramService logs: OSLogStore scoped to current process

- **Location**: `macos/Engram/Core/OSLogReader.swift:43`
- **Found by**: read-path-concurrency (round 1) — confidence high

> Evidence: `private static func makeStore() throws -> OSLogStore {

        return try OSLogStore(scope: .currentProcessIdentifier)
...

**Recommendation**: Service-process os_log entries are not readable from the app without elevated scope. #### H04. Error count is permanently 0: os_log .error mapped to "warn" and nothing ever logs .fault

- **Location**: `macos/Engram/Core/OSLogReader.swift:34`
- **Found by**: read-path-concurrency (round 1) — confidence high
- **Severity**: high
- **Verification**: confirmed by 2 adversarial lenses

levelString maps OSLogEntryLog.Level.error to "warn" and reserves "error" for .fault. countErrors() filters level == "error", so it counts only .fault entries. Grep over macos/Engram, macos/EngramService and macos/Shared finds zero call sites logging at .fault/critical — EngramLogger.error() uses Logger.error which arrives as .error and is reclassified as "warn". Result: SystemHealthView's 'Errors (24h)' KPI and ErrorDashboardView's countErrors/errorsByModule are structurally always 0/empty, even while EngramLogger.error fires (e.g. 'HomeView load failed'). Combined with the process-scope issue this makes the whole error-observability surface inert.

> Evidence: `case .error: return "warn"
case .fault: return "error"
...
static func countErrors(hours: Double = 24) throws -> Int {
    try recentLogs(level: "error", hours: hours, limit: Int.max).entries.count
}`

**Recommendation**: Map .error -> "error" and .fault -> "error" (or count both in countErrors). Add a unit test that writes a Logger.error entry and asserts countErrors >= 1 so the mapping cannot silently regress again.

#### H05. Popover source-health always shows 0 active sources and "—": fractional-seconds ISO parser rejects every indexed_at value

- **Location**: `macos/Engram/Views/PopoverView.swift:240`
- **Found by**: read-path-concurrency (round 1) — confidence high
- **Severity**: high
- **Verification**: confirmed by 2 adversarial lenses
- **Merged duplicates**: ui-frontend#0

PopoverView.loadData computes the health summary from DatabaseManager.sourceStats().latestIndexed (MAX(indexed_at)) using an ISO8601DateFormatter configured with .withFractionalSeconds and no fallback. ISO8601DateFormatter with .withFractionalSeconds fails on inputs WITHOUT fractional seconds. indexed_at never has fractional seconds: the live DB contains '2026-06-08T16:12:54Z' (Swift indexer) and '2026-03-06 02:22:48' (schema default datetime('now') per EngramMigrations.swift:26). Both fail to parse, so `active` is always 0 and `latest` is always nil → the popover permanently shows 0 active sources and last-indexed '—' regardless of real indexing activity. The writer itself (SessionSnapshotWriter.parseDate) uses a fractional-then-plain fallback chain; this view lacks it.

> Evidence: `let fmt = ISO8601DateFormatter()
fmt.formatOptions = [.withInternetDateTime, .withFractionalSeconds]
let active = stats.filter { s in
    guard !s.latestIndexed.isEmpty, let d = fmt.date(from: s.latestIndexed) else { return false }`

**Recommendation**: Parse with a fallback chain like Theme.formatTimestamp (strict fractional formatter, then plain ISO8601DateFormatter, then SQLite 'yyyy-MM-dd HH:mm:ss'), or centralize one shared lenient parser. Add a test feeding both observed indexed_at formats and asserting non-zero active count.

#### H06. Home 'Today' load executes six history-wide FTS searches, including two full scans of the 832 MB FTS table

- **Location**: `macos/Engram/Views/Pages/HomeView.swift:528`
- **Found by**: read-path-concurrency (round 1) — confidence high
- **Severity**: high (downgraded from medium during verification)
- **Verification**: confirmed by 1 adversarial lens
- **Merged duplicates**: ui-frontend#8

loadTodayFollowUps loops TodayFollowUps.queries = ["follow-up", "followup", "deferred", "remaining", "延后", "跟进"] and runs db.searchWithSnippets for each on every HomeView .task load. The two CJK queries take the LIKE path — `f.content LIKE ?` over sessions_fts (180,283 rows, DB 832 MB on the reviewer's machine): two full content scans per Home open. > Evidence: `for query in TodayFollowUps.queries {

**Recommendation**: Push the 72h window into SQL (searchWithSnippets already accepts `since:` — pass it) so the LIKE/FTS scans are bounded, and/or query the recent-session window first and run keyword matching over only those rows. Consider caching the result per app session.

#### H07. Sessions modified more than 2 days ago are never ingested after the one-time full scan

- **Location**: `macos/EngramService/Core/EngramServiceRunner.swift:371`
- **Found by**: adapters (round 1) — confidence high
- **Severity**: high
- **Verification**: confirmed by 2 adversarial lenses

The startup scan uses the full adapter set only when the usage_parser_version metadata key (currently "4") is missing or stale; otherwise it uses recentActiveAdapters(), which filters every file-backed source to files modified in the last 2 days (SessionAdapterFactory.recentActiveAdapters, days: 2) and limits Codex to today/yesterday date directories. The periodic loop (runIndexingLoop, 5-minute interval) also only calls writer.indexRecentSessions() with the same 2-day window. The FSEvents watcher is explicitly not wired into the runtime (comment in EngramDatabaseIndexer.swift:74-78: "the FSEvents watcher is not wired into the runtime"), and SessionWatcher/NonWatchableSourceRescanner are referenced only from EngramCoreTests. Consequence: if the service is not running for more than 2 days (machine off, app quit, user pauses Engram) while AI sessions accumulate, those sessions' mtimes fall outside the window by next launch and they are silently never indexed — there is no full-rescan command, and the next full scan happens only when developers bump usage_parser_version or the DB is rebuilt.

> Evidence: `let startupAdapters = usageParserBackfillNeeded
    ? defaultAdapters
    : SessionAdapterFactory.recentActiveAdapters()
-- and EngramDatabaseIndexer.swift:76-78 --
/// false, and the FSEvents watcher is not wired into the runtime, so
/// without this the only live path leaves new children top-level until the
/// next restart.`

**Recommendation**: Add a periodic or on-launch reconciliation full scan (e.g. daily, or whenever lastSuccessfulScanAt is older than the recency window), or persist the last scan timestamp and use it instead of a fixed now-2d cutoff for modifiedSince. Alternatively wire NonWatchableSourceRescanner/SessionWatcher into EngramServiceRunner.

#### H08. Kimi token usage massively undercounted: per-turn StatusUpdate overwrite plus turn-index drift

- **Location**: `macos/Shared/EngramCore/Adapters/Sources/KimiAdapter.swift:232`
- **Found by**: adapters (round 1) — confidence high
- **Severity**: high
- **Verification**: confirmed by 2 adversarial lenses

readTurnMetadata overwrites the current turn's usage with each StatusUpdate (`turns[turns.count - 1].usage = usage`) instead of accumulating. Verified against a real session (~/.kimi/sessions/01a01.../1a1aa9c9...): one TurnBegin contains 12 StatusUpdates, each a per-API-call delta (e.g. input_other 2928, 465, 568, 16328, ... 10574); only the last delta survives. > Evidence: `} else if type == "StatusUpdate",

**Recommendation**: Accumulate StatusUpdate deltas per turn with mergeUsage instead of overwriting, and attach turn usage to the turn as a whole (or to the last assistant message of the turn) rather than via the per-message drifting turnIndex. - **Location**: `macos/Shared/EngramCore/Adapters/Sources/CodexAdapter.swift:67`
- **Found by**: adapters (round 1) — confidence high
- **Severity**: high
- **Verification**: confirmed by 2 adversarial lenses

**Recommendation**: Degrade instead of dropping: index the first maxMessages records with a truncated flag, and treat lineTooLarge/invalidUtf8 as per-line skips (count them, don't fail the file). Surface a per-session 'partially indexed' status instead of an os_log-only error. Add tests for a >10k-line fixture and a fixture containing one oversized line.

#### H11. Manual unlink does not reset tier — misclassified 'skip' session disappears from the entire UI after user unlinks it

- **Location**: `macos/EngramService/Core/EngramServiceCommandHandler.swift:553`
- **Found by**: grouping-tiering (round 1) — confidence high
- **Severity**: high
- **Verification**: confirmed by 2 adversarial lenses

The TS reference clearParentSession (src/core/db/parent-link-repo.ts:60-71) explicitly resets `tier = NULL` on manual unlink ('Clear the confirmed parent... and resets tier to NULL'), and the documented invariant says unlinked children get tier reset to NULL for re-evaluation. The Swift service handler's clearParentSession omits the tier reset entirely. Auto-classified children (polycli/dispatched paths set `tier = 'skip'` plus a confirmed link) that a user explicitly unlinks via SessionDetailView.unlinkParent (macos/Engram/Views/SessionDetailView.swift:602 -> serviceClient.clearParentSession) keep tier='skip' and agent_role='dispatched'. All list/search queries filter `(tier IS NULL OR tier != 'skip')` (e.g. macos/Engram/Core/Database.swift:161), so the session the user just rescued from a wrong grouping vanishes from every surface instead of reappearing top-level. The 'skip' tier is also pinned forever by the indexer upsert rule (sessions.tier='skip' AND agent_role IS NOT NULL preserves skip), so no re-index recovers it.

> Evidence: `UPDATE sessions
SET parent_session_id = NULL,
    link_source = 'manual',
    link_checked_at = datetime('now')
WHERE id = ?`

**Recommendation**: Match the TS semantics: in clearParentSession add `tier = CASE WHEN agent_role = 'subagent' THEN 'skip' ELSE NULL END` (preserving the subagent invariant like the cascade trigger does). Add a service-handler test: classify a session dispatched/skip, link it, unlink it, assert tier is NULL and it reappears in tier-filtered queries.

#### H12. Compensation unconditionally reverses the physical move even when the move never happened, displacing a pre-existing destination directory

- **Location**: `macos/EngramCoreWrite/ProjectMove/Orchestrator.swift:891`
- **Found by**: project-move (round 1) — confidence high
- **Severity**: high
- **Verification**: confirmed by 2 adversarial lenses

The catch block only skips compensation for DirCollisionError/SharedEncodingCollisionError ('preflightFailure'). Any other throw — including Step 1 (SafeMoveDir) failing because src does not exist — runs compensate(), whose step 5 always executes SafeMoveDir.run(src: attemptedDst, dst: originalSrc) with no flag tracking whether the forward move actually occurred. If the user typos src (or swaps src/dst) so that src is missing and dst is an existing unrelated directory, the forward move throws ENOENT, then compensation's reverse move passes SafeMoveDir's preflight (dst-of-reverse = src doesn't exist) and physically relocates the unrelated directory at dst to the typo'd src path, while the migration is logged as failed. The same mechanism fires on a double-undo race: after undo A restores oldPath, undo B's pipeline fails at Step 1 (newPath gone) and compensation moves the freshly restored oldPath back to newPath, diverging FS from DB. OrchestratorTests explicitly defers FS-compensation testing ('To exercise the FS compensate path, we'd need to inject a failing FsOpsHooks; defer that to a more elaborate spec'), so this path has no coverage.

> Evidence: `// 5. Reverse the physical move.
    do {
        _ = try SafeMoveDir.run(src: attemptedDst, dst: originalSrc)
        report.moveReverted = true`

**Recommendation**: Track whether Step 1 succeeded (e.g. an Optional<MoveResult> set after SafeMoveDir.run returns) and pass it to compensate(); only attempt the physical move-back when the forward move actually executed. Add an FsOpsHooks-injected test for 'src missing + dst exists' asserting the dst directory is untouched.

#### H13. Gemini projects.json registry is not updated when the slug rename is a noop (same-basename move) or the tmp dir is missing

- **Location**: `macos/EngramCoreWrite/ProjectMove/Orchestrator.swift:456`
- **Found by**: project-move (round 1) — confidence high
- **Severity**: high
- **Verification**: confirmed by 2 adversarial lenses

Step 2.5 applies the projects.json rewrite only when a gemini-cli dir rename actually happened. encodeGemini slugifies only the path basename, so moving /a/foo → /b/foo produces identical slugs; the loop records the gemini plan as skippedDirs(.noop) and never adds a DirRenamePlan, so GeminiProjectsJSON.apply is skipped. But the registry is keyed by absolute cwd ('maps absoluteCwd → projectBasename' per the module header), so after a same-basename move (the common reorganize-parent-dir case) projects.json still maps the OLD cwd and lacks the new cwd. The module's own doc states the consequence: 'If we rename the tmp dir without updating the JSON, sessions detach silently.' The same gap applies when the tmp dir is missing but a registry entry exists. The generic JSONL sweep cannot fix this because projects.json lives at ~/.gemini/projects.json, outside the scanned ~/.gemini/tmp root.

> Evidence: `// Step 2.5: apply Gemini projects.json rewrite (after the dir rename).
            if let plan = geminiProjectsPlan,
               renamedDirs.contains(where: { $0.sourceId == .geminiCli }) {
                try GeminiProjectsJSON.apply(plan: plan)
                geminiProjectsApplied = true
            }`

**Recommendation**: Apply the projects.json plan whenever plan.oldEntry exists (the cwd key changed), independent of whether the tmp dir itself was renamed. Gate only on dryRun. Add a test moving /parent1/name → /parent2/name asserting the registry key is rewritten.
