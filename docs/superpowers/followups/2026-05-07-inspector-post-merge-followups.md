## 1. Deterministic MCP sqlite fixture

**Current evidence**

- `tests/fixtures/mcp-contract.sqlite` is regenerated by `npm run generate:mcp-contract-fixtures` (`scripts/gen-mcp-contract-fixtures.ts`).
- Two consecutive runs against the same input produce different bytes for the sqlite file (page layout / freelist / rowid encoding drift) even though every JSON golden the contract tests consume is byte-identical between runs.
- Acknowledged in the inspector PR description and in slice 7 verification notes.

**Current impact**

- JSON goldens are stable; hermeticity holds (no `mcp-inspector-*` rows leak into the shared sqlite, no `/Users/bing` paths, no secrets — verified via `sqlite3 ... "select id from sessions where id like 'mcp-inspector%';"` and `rg "/Users/bing|Authorization|sk-|AIza" tests/fixtures/mcp-golden`).
- Reviewers regenerating the fixture locally will see `mcp-contract.sqlite` dirty in `git status`, which is noisy but not incorrect.
- Pre-existing — not introduced by the inspector harness. Does not block merge.

**Suggested next slice**

Pick one of:

- Make the generator emit a deterministic sqlite (e.g. `VACUUM INTO`, fixed page size, sorted insert order, deterministic rowids, `PRAGMA journal_mode=DELETE` final pass), then re-baseline the committed binary and add a determinism assertion to the generator script.
- Or change the test/fixture strategy so the binary is not committed at all (re-derive on demand from the JSON goldens or from a deterministic seed in CI), and document the new flow in `tests/fixtures/mcp-golden/README.md`.

**Owner decision**

- Pick approach (deterministic-binary vs. don't-commit-binary).
- Confirm there are no downstream consumers (perf baselines, Swift schema-compat checks) that depend on the exact committed bytes — `docs/swift-single-stack/baseline-inventory.md` and `docs/performance/baselines/2026-04-23-node-runtime-baseline.json` reference this fixture path; verify they only need the JSON contract or the schema, not the byte-identical binary.

## 2. Signing configuration portability

**Current evidence**

- Inspector slice 7 added signing settings to the `EngramTests` target in `macos/project.yml` so the xctest bundle can dlopen into the hardened-runtime host:
  - `CODE_SIGN_IDENTITY: "Apple Development"`
  - `CODE_SIGN_STYLE: Automatic`
  - `DEVELOPMENT_TEAM: J25GS8J4XM`
- The host `Engram` target already used the same `J25GS8J4XM` team. Other test targets (`EngramServiceCoreTests`, `EngramCoreTests`) keep `CODE_SIGNING_ALLOWED: NO`.
- Auto-memory `macos-signing.md` and `2026-05-07-app-review-remediation-closeout.md` already document the team binding for release export (`ENGRAM_TEAM_ID=J25GS8J4XM macos/scripts/build-release.sh`).

**Current impact**

- Works for this repo and this Apple Developer team; verified by `xcodebuild ... -only-testing:EngramTests/ThemeTests` and `EngramTests/EngramServiceClientTests` (13/13 pass) on the dev machine.
- Anyone forking the repo with a different team has to edit `macos/project.yml` (mirror the host target's signing block onto `EngramTests`) before the test bundle can dlopen. Not blocking, but it is a friction point and the failure mode (Team ID mismatch dlopen rejection) is opaque if you have not seen it before.
- Public fork policy + branch policy are tracked in auto-memory `2026-04-29-open-source-publication.md`.

**Suggested next slice**

Pick one of:

- Move signing identity into an `xcconfig` (e.g. `macos/Configs/Signing.xcconfig`) referenced from `project.yml`, with a `Signing.local.xcconfig` override in `.gitignore`. Document the override flow in the macOS section of the project README / CLAUDE.md.
- Or thread an `ENGRAM_TEAM_ID` (and optional `ENGRAM_CODE_SIGN_IDENTITY`) environment variable through `project.yml` via xcodegen variable substitution, mirroring how `macos/scripts/build-release.sh` already accepts `ENGRAM_TEAM_ID`.
- Or, lowest-effort: add a short "Forking / different Apple Developer team" section to the macOS README documenting the exact lines to change, and surface the same hint as a comment in `macos/project.yml` next to the `EngramTests` block.

**Owner decision**

- Pick the level of investment (docs only vs. xcconfig vs. env-var threading).
- Confirm whether the public fork story is "fork + edit `project.yml`" (current) or "fork + set env var" (preferred for downstream contributors).

## 3. Optional future bridge revisit (Option C)

**Current evidence**

- Option C — re-bundling a Node/HTTP bridge inside the `.app` so Swift can call the TypeScript `SessionInspector` derivation — was explicitly **not** adopted. Option A (Swift-native parity) shipped instead.
- Decision history is preserved in `docs/superpowers/plans/2026-05-07-llm-session-inspector-harness.md` (Task 5 section, "Future work (out of scope)") and in `docs/superpowers/specs/2026-05-07-llm-session-inspector-harness-design.md` (Bridge Note rewritten to "Active UI surface").
- Result: TypeScript `SessionInspector` (in `src/core/session-inspector.ts`) and Swift `EngramServiceSessionInspector` (in `macos/Shared/Service/EngramServiceModels.swift` + `SQLiteEngramServiceReadProvider.inspectSession`) are deliberately parallel implementations.
- Drift is currently caught by:
  - The Swift parity test `EngramServiceCoreTests/EngramServiceInspectorTests/testDecodesContractGoldenFixture`, which decodes `tests/fixtures/mcp-golden/session_inspector.fixture.json` directly into the Swift DTO.
  - The TypeScript harness tests (`tests/core/llm-inspector-harness.test.ts`) that pin every observable shape on the TS side.
  - The MCP golden fixture, which is the single source of truth shared by both sides.

**Current impact**

- No runtime impact. Cost is the duplicated derivation logic across TS and Swift, plus the mental overhead of remembering that any DTO change has to land in lockstep on both sides + the golden fixture.
- Contract-golden parity test fails fast if either side drifts.

**Suggested next slice**

If a future milestone justifies re-sharing derivation between TS and Swift, treat it as a brand-new slice — for example **"Service Bridge to TypeScript Inspector"** — that explicitly:

- Re-introduces a Node bundle inside the `.app` (or an alternative shared-derivation strategy such as a Rust core, a precompiled WASM module, or a sidecar gRPC service).
- Re-evaluates Stage 5 single-stack constraints (auto-memory `macos-app-architecture.md` records that "TS `src/` does NOT ship inside the .app").
- Re-runs the full Phase 0 invariant matrix (no provider calls, no transcript streaming, no schema/migration changes, no `which`/`Process()`/`URLSession`/`fetch` in inspector code) for whatever new bridge shape ships.

**Owner decision**

- This is dormant by default. Only revisit if duplicated derivation actually causes a real bug or a real maintenance cost. If it does come back, the gating decision is whether to re-bundle Node (Option C) or to pick a different shared-derivation approach. Either path requires explicit re-approval before any work starts — not a follow-up that should silently grow into a refactor.
