# Competitive Session Improvements Design

Date: 2026-07-02
Status: Implemented in branch `codex-provider-audit-remediation`
Primary owner: Engram maintainers
Reviewers: Claude, Codex
Related plan: `docs/superpowers/plans/2026-07-02-competitive-session-improvements.md`

## Context

Engram already has the strongest foundation among the compared tools for cross-tool
AI-session continuity: a native Swift product path, Swift MCP helper, broad source
coverage, normalized session metadata, parent/child linking, project aliases,
project migration commands, and cost/tool/file analytics. The competitive review
against Agent Sessions 4.0 and ReadOut identified several improvements that fit
Engram's existing architecture without changing its product identity.

This spec defines an ordered design for those improvements. It is intentionally
SPEC-only: it describes goals, boundaries, sequencing, ownership, acceptance, and
validation. It does not implement product code.

## Problem

Engram's current strengths are under-expressed in the product and documentation:

- Runtime facts such as source/tool counts can drift between docs and code.
- Engram's positioning is less immediate than Agent Sessions' history/runway
  story or ReadOut's dashboard story.
- Existing parent/child session data can be clearer in the UI through stable
  badges, filters, and hierarchy states.
- Project continuity features such as aliases and migration safety are stronger
  through MCP/service APIs than through visible app surfaces.
- Cost, quota, archived-session, side-chat, and large-history performance work
  needs a sequenced roadmap so it does not become a broad dashboard rewrite.

## Goals

1. Make Engram's shipped runtime capabilities auditable from one source of truth.
2. Clarify Engram's positioning as a cross-tool memory and project-continuity
   layer that current AI assistants can query through MCP.
3. Productize existing session hierarchy metadata with visible, filterable UI
   concepts without changing storage semantics unnecessarily.
4. Surface project continuity and usage/cost information in focused native app
   panels that reinforce Engram's MCP-first value.
5. Add read-only coverage checks for competitive session shapes, especially
   Codex side chats and archived sessions, before any restore/write-back design.
6. Preserve local-first privacy, Swift product ownership, and service security
   boundaries.
7. Keep every delivery phase small enough to review independently.

## Non-Goals

The following are explicitly out of scope for this spec and for every phase
below:

- ReadOut's "card-as-answer" pattern.
- Ask Engram.
- Chat-style Q&A over sessions.
- LLM-generated answer UX.
- A general assistant embedded in the app.
- Any new product surface where prose generated by a model is the primary answer.
- Product startup paths that shell out to Node or depend on TypeScript runtime
  code.
- Reintroducing Node schema compatibility gates or Node bundle checks.
- Destructive restore/write-back behavior for archived sessions or side chats.

Negative acceptance criterion: after every phase, reviewers must confirm that no
Ask Engram, chat Q&A, card-as-answer, or LLM answer-generation entry point was
introduced.

## Scope

### In scope

- Documentation and runtime capability reporting.
- Swift MCP tool/source registry truth surfacing.
- SwiftUI surfaces that display existing or newly indexed metadata.
- EngramService and EngramCoreRead/Write boundaries where needed for shipped
  Swift product behavior.
- Read-only MCP behavior for diagnostics and project continuity where it already
  fits Engram's tool model.
- Focused Swift tests and fixture/parity tests for behavior changes.
- Documentation updates for privacy, MCP tools, and user-facing behavior.

### Out of scope unless a later spec reopens them

- TypeScript as shipped runtime.
- Semantic session search as a shipped Swift feature.
- Peer sync.
- Default network calls for quota/runway.
- Automatic credential discovery.
- Restore/write-back for archived sessions.
- Broad visual redesign of the app.

## Design Principles

1. **Runtime truth over copied numbers.** Dynamic counts and capability lists
   should be derived from Swift registries or tested generated docs, not hand
   repeated across pages.
2. **MCP-first differentiation.** Human UI improvements must reinforce the fact
   that Engram makes prior work available to the current AI assistant.
3. **Read-only first.** New competitive coverage begins as detection, display,
   filtering, and diagnostics before any mutating workflow is considered.
4. **No hidden network.** Quota/runway and cost surfaces must not contact
   provider services unless the user explicitly configures and triggers that
   capability.
5. **Swift product path only.** Product behavior belongs in the Swift app,
   service, MCP helper, and shared Swift modules. TypeScript remains reference,
   fixture, or development material.
6. **Small reviewable increments.** Each phase must leave the product in a
   coherent state and avoid cross-phase partial behavior.

## Target Capability Sequence

### Phase 0 - SPEC and review gate

Purpose: agree on this design before implementation starts.

Acceptance criteria:

- This spec and its companion plan exist under `docs/superpowers/`.
- Codex review has no blocking findings, or findings are resolved in the docs.
- The explicit non-goals above remain present.
- No product code is changed in this phase.

### Phase 1 - Runtime capability truth and positioning

Purpose: prevent docs/runtime drift and make Engram's current value obvious.

Scope:

- Define a single source-of-truth contract for source count, MCP tool count, and
  unavailable/disabled runtime features.
- Document how README and MCP docs should consume that truth.
- Define the user-facing positioning statement: Engram lets the current AI query
  cross-tool history, rules, memories, and project continuity through MCP.

Acceptance criteria:

- Maintainers can verify source/tool counts without manually reading multiple
  files.
- README and MCP docs describe the same capability set.
- Disabled or degraded features are named explicitly, including keyword-only
  session search when semantic session search is unavailable.
- No Ask Engram or answer-generation language appears in positioning copy.

### Phase 2 - Session taxonomy, badges, and filters

Purpose: make existing hierarchy and session-shape metadata visible without
changing the core parent-link model.

Scope:

- Define stable taxonomy labels for `subagent`, `workflow`, `side`, `archived`,
  `orphan`, and `suggested parent` states.
- Specify where badges and filters appear in Sessions, Timeline, Search, and
  session detail surfaces.
- Preserve current tiering behavior: subagent content remains accessed through
  parent sessions unless a separate approved design changes that rule.

Acceptance criteria:

- Users can identify and filter noisy or nested session types.
- Existing parent/child grouping behavior is not weakened.
- Suggested parent state is visible as advisory, not confirmed truth.
- No storage migration is required unless a later implementation plan proves an
  existing field cannot represent the taxonomy.

### Phase 3 - Project continuity surfaces

Purpose: expose Engram's project-alias and migration-safety strengths in the app.

Scope:

- Define native UI surfaces for viewing project aliases, reviewing alias effects,
  and understanding move/archive/recover state.
- Keep writes routed through EngramService commands; app and MCP clients do not
  gain direct SQLite write paths.
- Make destructive project actions explicit and confirmable.

Acceptance criteria:

- Users can inspect aliases and project-continuity state from the app.
- Migration/recovery state is understandable without reading logs or using MCP
  directly.
- Every mutating action remains service-gated and confirmable.
- Privacy docs continue to state local file and database behavior accurately.

### Phase 4 - Usage, cost, and optional runway panels

Purpose: improve daily operational value without creating a dashboard clone.

Scope:

- Define focused panels for model spend, token usage, cache behavior when
  available, tool/file activity, and high-cost sessions/projects.
- Define an optional, read-only runway concept for provider quotas where data is
  locally available or explicitly configured.
- Avoid implicit credential probing and default provider-network calls.

Acceptance criteria:

- Users can answer basic usage questions from existing indexed data.
- Any quota/runway data source is labeled with freshness and source.
- Missing quota data degrades to an explanatory empty state, not an error.
- No provider credentials are read unless the user explicitly configures that
  path in a later approved implementation.

### Phase 5 - Competitive coverage and performance audit

Purpose: close known session-shape gaps and avoid large-history regressions.

Scope:

- Specify fixture coverage for Codex side chats and archived-session shapes.
- Specify read-only detection and filtering before restore/write-back.
- Define a performance audit for Sessions, Timeline, and Search list rendering:
  list rows should use indexed lightweight state, not full transcript parsing.

Acceptance criteria:

- Coverage gaps are represented as tests or documented unsupported cases.
- Archived and side-chat content, if detected, is read-only and clearly labeled.
- Large-history surfaces have explicit performance expectations and regression
  tests or manual verification steps.
- Restore/write-back remains out of scope.

### Phase 6 - Focused visual polish and accessibility

Purpose: apply polish only after data semantics are stable.

Scope:

- Define consistent KPI rows, status badges, empty states, loading states, and
  accessible labels for the surfaces above.
- Keep polish tied to existing Engram screens; do not introduce a new assistant
  or answer surface.

Acceptance criteria:

- The improved surfaces are understandable in empty, loading, error, and normal
  states.
- Accessibility labels and keyboard navigation expectations are documented for
  interactive controls.
- Visual changes do not expand feature scope.

## Ownership and Review Gates

- Product/spec owner: keeps scope, non-goals, and sequencing coherent.
- Swift UI owner: owns app surface design, accessibility requirements, and manual
  verification flows.
- Service/data owner: owns EngramService, indexing, migration/backfill, and data
  lifecycle requirements when a phase touches them.
- MCP/docs owner: owns MCP tool documentation, runtime capability docs, and
  compatibility notes.
- QA/review owner: owns validation matrix, fixture expectations, and regression
  evidence.
- Codex reviewer: reviews each SPEC/plan update or implementation PR for scope
  creep, acceptance clarity, and security/privacy regressions.

No implementation phase should begin until its owner and validation evidence are
named in the companion plan.

## Validation Matrix

Validation is triggered by touched area rather than run unconditionally:

| Touched area | Required validation |
| --- | --- |
| Docs/spec only | Markdown review, link/path validation, Codex review |
| SwiftUI product surface | Focused Swift tests where applicable, Xcode build, manual app flow |
| Service/indexing/storage | Focused Swift unit/parity tests, migration/backfill verification, rollback notes |
| MCP tool behavior | MCP executable/tool tests and docs update |
| TypeScript reference tooling | `npm run build`, `npm test`, `npm run lint` only for touched TS paths |
| Security/privacy boundary | Explicit review of service token, peer UID, local binding, Host/CORS, bearer auth, body caps, and path normalization as applicable |

## Privacy and Security Requirements

- Default behavior remains local-first and zero-telemetry.
- Any external provider call, AI call, or offload path must be explicit in UI and
  docs before implementation.
- EngramService remains the authority for writes.
- App and MCP clients must not add direct SQLite write paths.
- HTTP/API behavior is unchanged by this spec; if a later phase touches it, Host,
  CORS, bearer auth, request-body caps, and filesystem path normalization must be
  revalidated.
- Local service token and peer-UID boundaries remain unchanged unless a separate
  security-reviewed design changes them.

## Rollback and Failure Rules

- Docs-only changes roll back by reverting the documentation PR.
- Read-only UI changes must be disableable by removing or hiding the surface
  without changing indexed data.
- Any future storage/indexing phase must define forward migration, failed-backfill
  behavior, and data-loss prevention before implementation.
- If a phase cannot meet its negative acceptance criterion against Ask Engram,
  card-as-answer, or answer-generation UX, it stops and returns to design review.

## Open Questions

1. Should runtime capability truth be generated into docs, exposed through a
   diagnostic command, or both?
2. Which existing fields can represent `side` and `archived` taxonomy without a
   schema change?
3. Which quota/runway data can be derived locally without provider-network calls?
4. What minimum large-history dataset should become the performance baseline?
5. Which app surface should own project alias management: Projects, Settings, or
   a project detail sheet?
